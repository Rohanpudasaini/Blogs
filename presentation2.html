What is Atomicity?

"An atomic transaction is an indivisible and irreducible series of database operations such that either all occur, or none occur."
— C. J. Date, Database Systems Expert
✅

All-or-Nothing

Either all operations succeed completely, or none are applied

🔄

Rollback on Failure

If any part fails, the entire transaction is rolled back

Transaction Fundamentals

A transaction represents a logical unit of work consisting of one or more database operations.

INSERT operations
UPDATE operations
DELETE operations
SELECT operations 
-- Example transaction
BEGIN TRANSACTION;

UPDATE accounts
SET balance = balance - 100
WHERE account_id = 1;

UPDATE accounts
SET balance = balance + 100
WHERE account_id = 2;

INSERT INTO transactions
(from_account, to_account, amount)
VALUES (1, 2, 100);

COMMIT;

The Problem with Non-Atomic Commits

Failure Scenario:

-- Money transfer transaction
BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 500 WHERE account_id = 'A';
UPDATE accounts SET balance = balance + 500 WHERE account_id = 'B';
COMMIT;
⚠️ System crashes after first UPDATE
💸 Account A loses $500
❌ Account B never receives $500
🔄 Database in inconsistent state

Durability

Changes survive crashes

🔄

Recovery

Complete system recovery

⚡

Performance

Sequential writes

WAL Protocol Fundamentals

The WAL Process:

1
Log changes to WAL file first
2
Flush log to durable storage
3
Apply changes to database
4
Mark transaction as committed
8 / 16

WAL File Structure

Key Components:

Transaction ID

Unique identifier for each transaction

Operation Type

INSERT, UPDATE, DELETE, COMMIT

Before/After Images

Original and modified data for rollback/recovery

Checksum

CRC-32 for data integrity verification

-- WAL Record Structure
struct WALRecord {
    uint32_t transaction_id;      -- Unique identifier
    uint32_t record_type;         -- INSERT, UPDATE, DELETE, COMMIT
    uint32_t table_id;            -- Which table is modified
    uint64_t row_id;              -- Primary key of affected row
    uint32_t data_length;         -- Length of data payload
    char* before_image;           -- Original row data (UNDO)
    char* after_image;            -- Modified row data (REDO)
    uint32_t checksum;            -- CRC-32 integrity check
    uint64_t timestamp;           -- When operation occurred
};

Data Integrity Protection

CRC-32 Checksum Protection:

✅ Corruption Detection

Identifies any bit corruption in log records

✅ Partial Write Detection

Discards incomplete records from crashes

✅ Data Consistency

Ensures only valid records are applied

// Simplified CRC-32 calculation
uint32_t calculate_crc32(const char* data, size_t length) {
    uint32_t crc = 0xFFFFFFFF;
    for (size_t i = 0; i < length; i++) {
        crc = crc32_table[(crc ^ data[i]) & 0xFF] ^ (crc >> 8);
    }
    return crc ^ 0xFFFFFFFF;
}

Log File Organization

Modern WAL Architecture:

File Size Optimization

16MB to several GB depending on database system

Page-Based Architecture

4KB (SQLite) or 8KB (PostgreSQL) page sizes

Log Sequence Numbers

Unique identifiers combining file number and offset

// Log Sequence Number structure
struct LSN {
    uint32_t file_number;    -- WAL file number
    uint32_t page_offset;    -- Offset within file
    uint32_t record_offset;  -- Offset within page
};

// Benefits
- O(1) append operations
- Efficient recovery
- Parallel processing
- Point-in-time recovery

Recovery Process

System Recovery Sequence:

1
Scan WAL files from last checkpoint
2
Validate checksums and discard corrupted records
3
Replay REDO operations for committed transactions
4
Rollback UNDO operations for incomplete transactions
